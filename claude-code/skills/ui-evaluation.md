# UI評価スキル

このスキルは、UIデザインを多角的な視点から評価し、改善点を発見するためのフレームワークを提供します。

## 基本理念

**「ユーザーが迷わず、疲れず、目的を達成できるUI」**

良いUIとは、ユーザーが意識せずに使えるUIです。評価を通じて摩擦を発見し、取り除きましょう。

## 1. デザインの4原則（CRAP）

Robin Williamsが提唱したデザインの基本原則です。

### 1.1 近接（Proximity）

**関連する要素を近くに、関連しない要素を離す**

| 評価項目 | チェック |
|---------|---------|
| 関連する情報がグループ化されているか | ☐ |
| グループ間に十分な余白があるか | ☐ |
| ラベルと入力欄が近接しているか | ☐ |
| 論理的な情報のまとまりがあるか | ☐ |

```
❌ 悪い例
┌─────────────────────────┐
│ 名前                     │
│                         │
│ [入力欄]                 │
│                         │
│                         │
│ メール                   │
│                         │
│ [入力欄]                 │
└─────────────────────────┘
（ラベルと入力欄が離れすぎ）

✅ 良い例
┌─────────────────────────┐
│ 名前                     │
│ [入力欄]                 │
│                         │
│ メール                   │
│ [入力欄]                 │
└─────────────────────────┘
（ラベルと入力欄が近い）
```

### 1.2 整列（Alignment）

**すべての要素を意図的に配置する**

| 評価項目 | チェック |
|---------|---------|
| 要素が見えない線で揃っているか | ☐ |
| 左揃え/中央揃え/右揃えが一貫しているか | ☐ |
| グリッドシステムに沿っているか | ☐ |
| 「なんとなく」の配置がないか | ☐ |

```
❌ 悪い例
┌─────────────────────────┐
│    タイトル              │
│ 説明文がここにあります    │
│        [ボタン]          │
└─────────────────────────┘
（揃いがバラバラ）

✅ 良い例
┌─────────────────────────┐
│ タイトル                 │
│ 説明文がここにあります    │
│ [ボタン]                 │
└─────────────────────────┘
（左揃えで統一）
```

### 1.3 反復（Repetition）

**視覚的要素を一貫して繰り返す**

| 評価項目 | チェック |
|---------|---------|
| 同じ機能に同じスタイルが使われているか | ☐ |
| 色の使い方が一貫しているか | ☐ |
| フォントの使い方が統一されているか | ☐ |
| アイコンのスタイルが揃っているか | ☐ |
| 間隔・余白のパターンが統一されているか | ☐ |

```
❌ 悪い例
[保存] [Cancel] [削除する]
（ボタンのスタイルがバラバラ）

✅ 良い例
[保存] [キャンセル] [削除]
（統一されたスタイルと文言）
```

### 1.4 コントラスト（Contrast）

**異なる要素は明確に異ならせる**

| 評価項目 | チェック |
|---------|---------|
| 見出しと本文の差が明確か | ☐ |
| 重要な要素が目立っているか | ☐ |
| CTAボタンが他のボタンと区別できるか | ☐ |
| 中途半端な違いになっていないか | ☐ |
| 視覚的階層が明確か | ☐ |

```
❌ 悪い例
見出し（16px, 濃いグレー）
本文（15px, グレー）
（差が曖昧で区別しにくい）

✅ 良い例
見出し（24px, 黒, 太字）
本文（16px, グレー, 標準）
（明確な差がある）
```

### 2.5 眼球運動の生理学的特性

人間の眼球運動には生理学的な特性があり、UI配置に影響を与えます。

#### 外眼筋の構造

人間の眼球は6つの外眼筋によって制御されています。

**水平方向の運動**: 内側直筋と外側直筋の**1対のみ**で完全に制御される

> "Horizontal movements are controlled **entirely** by the medial and lateral rectus muscles"
>
> — [NCBI Bookshelf - The Actions and Innervation of Extraocular Muscles](https://www.ncbi.nlm.nih.gov/books/NBK10793/)

> "Medial and lateral rectus muscles have **only horizontal actions**... This is why these each only have one primary action and no secondary/tertiary actions"
>
> — [EyeWiki - Extraocular Muscles (American Academy of Ophthalmology)](https://eyewiki.org/Extraocular_Muscles)

**垂直方向の運動**: 上下直筋と上下斜筋の**2ペア（計4筋）**の協調動作が必要

> "Vertical movements require the **coordinated action** of the superior and inferior rectus muscles, **as well as** the oblique muscles"
>
> — [NCBI Bookshelf - The Actions and Innervation of Extraocular Muscles](https://www.ncbi.nlm.nih.gov/books/NBK10793/)

#### 水平方向と垂直方向の眼球運動の効率差

この構造的な違いにより、垂直方向のサッケード（急速眼球運動）は水平方向と比較して効率が低いことが研究で示されています。

> "**Vertical saccades are less efficient than horizontal saccades for all ages**"
>
> — [PubMed - Difference between vertical and horizontal saccades across the human lifespan (2018)](https://pubmed.ncbi.nlm.nih.gov/30171857/)

> "Vertical saccades had **significantly longer latencies** than horizontal saccades (**p = 0.0001**)"
>
> — [PubMed - Difference between vertical and horizontal saccades across the human lifespan (2018)](https://pubmed.ncbi.nlm.nih.gov/30171857/)

> "**Horizontal saccades are more rapid than vertical** and oblique saccades, due to a **higher performance of the horizontal recti muscles**"
>
> — [Behavior Research Methods - The saccade main sequence revised (2020)](https://link.springer.com/article/10.3758/s13428-020-01388-2)

具体的な速度差（[PLOS One, 2018](https://journals.plos.org/plosone/article?id=10.1371/journal.pone.0204008)）：
- 水平方向15°サッケード: 平均 **352°/s**
- 垂直方向10°サッケード: 平均 **303°/s**

#### UI設計への示唆

| 評価項目 | チェック |
|---------|---------|
| 参照情報と入力欄が水平方向に配置されているか | ☐ |
| 頻繁に比較する要素が横並びになっているか | ☐ |
| 縦方向の視線移動が最小限に抑えられているか | ☐ |
| 重要な情報が視線の自然な流れ（左→右）に沿っているか | ☐ |

##### 現実世界のアナロジー

テストの採点を想像してください。A4縦長の答案用紙を採点する際、多くの人は：
- **左に解答例**（参照情報）
- **右に生徒の答案**（確認・記入対象）

を横並びに置きます。これは無意識に**水平方向の視線移動が楽**であることを活用しています。

UIでも同様に、**「情報Aを参照しながら情報Bを入力・確認する」** 画面では、水平配置が有効です。

##### UI設計での適用例

```
❌ 垂直配置（参照と入力が上下 → 視線移動の負荷が高い）
┌─────────────────────────────────────┐
│ 【参照情報】                         │
│ 顧客名: 山田太郎                     │
│ 住所: 東京都渋谷区...                │
│ 電話: 03-xxxx-xxxx                  │
├─────────────────────────────────────┤
│ 【入力フォーム】                     │
│ 確認済み顧客名: [          ]        │
│ 確認済み住所:   [          ]        │
│ 確認済み電話:   [          ]        │
└─────────────────────────────────────┘
（参照のたびに視線を上下に大きく移動）

✅ 水平配置（参照と入力が左右 → 視線移動の負荷が低い）
┌─────────────────────┬─────────────────────┐
│ 【参照情報】         │ 【入力フォーム】     │
│                     │                     │
│ 顧客名: 山田太郎     │ 確認: [          ]  │
│ 住所: 東京都渋谷区.. │ 確認: [          ]  │
│ 電話: 03-xxxx-xxxx  │ 確認: [          ]  │
└─────────────────────┴─────────────────────┘
（左右の視線移動だけで参照と入力を往復できる）
```

##### 水平配置が特に有効な場面

- **データ入力画面**: 元データを見ながら新規入力
- **翻訳・校正画面**: 原文と訳文を並べて確認
- **比較画面**: Before/After、新旧バージョンの比較
- **レビュー画面**: 申請内容を見ながら承認・コメント入力

> **注意**: これは生理学的な観点からの示唆です。モバイル画面など横幅が限られる場合や、情報の論理構造上縦配置が適切な場合もあります。他の設計要素とのバランスを考慮してください。

## 3. 言葉の評価（UXライティング）

### 3.1 平易さ

| 評価項目 | チェック |
|---------|---------|
| 専門用語を避けているか | ☐ |
| 略語を使う場合は初出で説明しているか | ☐ |
| 中学生でも理解できる言葉か | ☐ |
| 一文が短いか（40文字以内推奨） | ☐ |

### 3.2 具体性

| 評価項目 | チェック |
|---------|---------|
| 曖昧な表現を避けているか | ☐ |
| 数値や具体例を示しているか | ☐ |
| ユーザーの次のアクションが明確か | ☐ |

```
❌ 曖昧な例
「エラーが発生しました」
「しばらくお待ちください」

✅ 具体的な例
「パスワードは8文字以上で入力してください」
「約3分で完了します」
```

### 3.3 一貫性

| 評価項目 | チェック |
|---------|---------|
| 同じ機能に同じ用語を使っているか | ☐ |
| ボタンラベルの文体が統一されているか | ☐ |
| 敬体/常体が統一されているか | ☐ |

### 3.4 トーン&ボイス

| 評価項目 | チェック |
|---------|---------|
| ブランドの人格に合った言葉遣いか | ☐ |
| 状況に応じた適切なトーンか | ☐ |
| 丁寧すぎず、粗すぎないか | ☐ |
| ユーザーを責めるような表現がないか | ☐ |

```
❌ ユーザーを責める例
「入力が間違っています」
「無効な操作です」

✅ ユーザーに寄り添う例
「メールアドレスの形式を確認してください」
「この操作を続けるには、ログインが必要です」
```

## 4. ニールセンの10ヒューリスティクス

### 4.7 柔軟性と効率性

| 評価項目 | チェック |

| ショートカットがあるか | ☐ |

| 評価項目 | チェック |
|---------|---------|

| 本当に必要な要素だけがあるか | ☐ |
| 余白が効果的に使われているか | ☐ |

### 4.9 エラーの認識・診断・回復の支援

| 評価項目 | チェック |
|---------|---------|
| エラーメッセージが平易な言葉か | ☐ |

| エラーコードだけで終わっていないか | ☐ |

```
❌ 悪いエラーメッセージ

✅ 良いエラーメッセージ
「サーバーに接続できません。しばらくしてから再度お試しください」

```

### 4.10 ヘルプとドキュメント

| 評価項目 | チェック |
|---------|---------|
