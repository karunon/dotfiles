# UXリサーチスキル

このスキルは、ユーザー中心設計を実現するためのUXリサーチを支援します。定性・定量調査の設計から実施、インサイト抽出、レポート作成までを包括的にカバーします。

## 基本理念

**「ユーザーを理解し、課題を発見し、価値ある解決策を導く」**

推測ではなくデータと観察に基づいた意思決定を行い、ユーザーの本質的なニーズを捉えます。

---

## 1. UXリサーチの全体プロセス

### 1.1 リサーチ計画

リサーチを始める前に、以下を明確にしてください：

| 項目 | 質問 |
|------|------|
| **目的** | 何を明らかにしたいのか？ |
| **仮説** | 現時点での想定は何か？ |
| **対象ユーザー** | 誰から学びたいのか？ |
| **手法** | どの調査手法が適切か？ |
| **成果物** | 調査結果をどう活用するか？ |

### 1.2 リサーチの種類と選択

| フェーズ | 目的 | 推奨手法 |
|---------|------|---------|
| **探索フェーズ** | ユーザー理解・課題発見 | インタビュー、エスノグラフィー、日記調査 |
| **定義フェーズ** | 要件整理・優先順位付け | カードソーティング、コンセプトテスト |
| **設計フェーズ** | 設計の検証・改善 | ユーザビリティテスト、A/Bテスト |
| **評価フェーズ** | 効果測定・継続改善 | アンケート、アナリティクス分析 |

## 2. ユーザーインタビュー

### 2.1 インタビュー設計の原則

#### 良い質問の特徴

```
✅ オープンエンド（自由回答形式）
✅ 具体的なエピソードを引き出す
✅ 「なぜ」を深掘りする
✅ 中立的で誘導しない

❌ はい/いいえで終わる質問
❌ 複数の質問を一度にする
❌ 専門用語を使う
❌ 答えを示唆する
```

#### 質問の構造（ファネル型）

```
1. ウォームアップ（信頼関係構築）
   └─ 背景・日常について軽く聞く

2. 広い質問（コンテキスト理解）
   └─ 「普段〜についてどのようにされていますか？」

3. 深い質問（詳細・エピソード）
   └─ 「最近〜したときのことを教えてください」
   └─ 「そのとき何を考えていましたか？」

4. 特定の質問（仮説検証）
   └─ 「〜の場面ではどうしていますか？」

5. クロージング
   └─ 「他に何か伝えたいことはありますか？」
```

### 2.2 インタビュースクリプトテンプレート

```markdown
## インタビュースクリプト

### 基本情報
- 調査目的: [目的を記載]
- 対象者条件: [リクルーティング条件]
- 所要時間: 約60分

### 導入（5分）
- 自己紹介と調査の目的説明
- 録音・録画の許可確認
- 「正解はない」ことの説明

### セクション1: 背景理解（10分）
1. 普段の[対象領域]について教えてください
2. [関連する行動]はどのくらいの頻度でされますか？

### セクション2: エピソード深掘り（25分）
3. 最近[対象行動]をしたときのことを詳しく教えてください
   - いつ、どこで？
   - 何がきっかけで？
   - どんな手順で？
   - 困ったことは？
4. そのときどんな気持ちでしたか？
5. なぜそのように[行動]したのですか？

### セクション3: 課題と要望（15分）
6. [対象領域]で困っていることはありますか？
7. 理想的には、どうなっていると嬉しいですか？

### クロージング（5分）
8. 今日お話しいただいた中で、特に大事だと思うことは？
9. 他に何か伝えたいことはありますか？
```

### 2.3 インタビュー中のテクニック

| テクニック | 説明 | 例 |
|-----------|------|-----|
| **沈黙を使う** | 相手が考える時間を与える | 3〜5秒待つ |
| **オウム返し** | 相手の言葉を繰り返して確認 | 「〜ということですね」 |
| **深掘り** | 抽象的な回答を具体化 | 「具体的には？」「例えば？」 |
| **感情を聞く** | 行動だけでなく感情を理解 | 「そのときどう感じましたか？」 |
| **5回のなぜ** | 根本原因を探る | 「それはなぜですか？」を繰り返す |

## 3. ユーザビリティテスト

### 3.1 テスト設計

#### テスト計画書テンプレート

```markdown
## ユーザビリティテスト計画

### 概要
- テスト対象: [プロダクト/機能名]
- テスト目的: [何を検証するか]
- 参加者数: 5〜8名（推奨）
- 所要時間: 1セッション60分

### 参加者条件
- [条件1: 例）週1回以上オンラインショッピングを利用]
- [条件2: 例）スマートフォンをメインで使用]
- 除外条件: [UI/UXの専門家、競合他社勤務など]

### タスク設計
| # | タスク | 成功基準 | 想定時間 |
|---|--------|---------|---------|
| 1 | [タスク内容] | [完了条件] | 3分 |
| 2 | [タスク内容] | [完了条件] | 5分 |

### 評価指標
- タスク成功率
- タスク完了時間
- エラー数
- SUS（System Usability Scale）スコア
- 主観的満足度（1-7段階）
```

#### タスク設計のポイント

```
✅ 現実的なシナリオに基づく
✅ ゴールは明確、手順は指定しない
✅ 適度な難易度（簡単→難しいの順）
✅ 独立したタスク（順序依存を避ける）

❌ UIの用語を使う（「〜ボタンを押して」）
❌ 手順を指示する（「まず〜して、次に〜」）
❌ 曖昧なゴール（「使ってみてください」）
```

### 3.2 テスト実施

#### モデレーターの心得

```markdown
## 基本姿勢
- 中立を保つ（「良いですね」「それは違います」と言わない）
- 参加者を責めない（「システムをテストしています」）
- 助けすぎない（困っても少し見守る）

## 発話プロトコル
- 「思ったことをそのまま声に出してください」
- 「正解はありません、感じたままに操作してください」

## 困ったときの対応
- 沈黙が続く → 「今何を考えていますか？」
- 助けを求められる → 「普段ならどうしますか？」
- 諦めそう → 「あと少しだけ試してみてください」
- 完全に詰まる → ヒントを出すが記録に残す
```

```markdown
## ペルソナ: [名前]

### 基本情報
- 年齢:
- 職業:
- 居住地:
- 家族構成:

### 写真
[代表的なイメージ写真]

### 背景ストーリー
[この人物の日常や価値観を2-3文で]

### ゴールと動機
**やりたいこと（Goals）**
1. [主要なゴール]
2. [副次的なゴール]

**その理由（Motivations）**
- [なぜそのゴールを持っているか]

### 課題と不満（Pain Points）
1. [最大の課題]
2. [日常的な不満]
3. [潜在的な課題]

### 行動パターン
- [関連する日常の行動]
- [情報収集の方法]
- [意思決定のプロセス]

### 利用デバイス・サービス
- メインデバイス:
- よく使うアプリ:
- 情報源:

### 代表的な発言（Quote）
> 「[このペルソナらしい発言]」
```

### 4.2 ペルソナ作成の注意点

```
✅ リサーチデータに基づく
✅ 行動と動機にフォーカス
✅ チームで共有・活用される
✅ 定期的に見直し・更新

❌ 想像で作る
❌ 属性情報だけ並べる
❌ 作って終わり
❌ 多すぎるペルソナ（3-5人が目安）
```

## 6. アンケート設計

### 6.1 アンケート設計の原則

#### 質問タイプの選択

| タイプ | 用途 | 例 |
|--------|------|-----|
| **単一選択** | 排他的な選択肢 | 性別、満足度 |
| **複数選択** | 複数該当する場合 | 利用機能、課題 |
| **リッカート尺度** | 態度・評価の測定 | 満足度（1-5） |
| **NPS** | 推奨度測定 | 0-10のスコア |
| **自由記述** | 深い理解が必要な場合 | 改善要望 |

#### 質問文の書き方

```
✅ 一つの質問で一つのことを聞く
✅ 具体的で明確な表現
✅ 中立的な言い回し
✅ 回答しやすい選択肢

❌ ダブルバーレル（2つのことを1問で聞く）
❌ 誘導質問
❌ 専門用語
❌ 否定形の質問
```

### 6.2 SUS（System Usability Scale）

```markdown
## SUSアンケート（10問）

以下の各項目について、1（全くそう思わない）〜5（強くそう思う）で回答してください。

1. このシステムを頻繁に使いたいと思う
2. このシステムは不必要に複雑だと思う
3. このシステムは使いやすいと思う
4. このシステムを使うには技術者のサポートが必要だと思う
5. このシステムの様々な機能がよく統合されていると思う
6. このシステムには一貫性がないと思う
7. 多くの人がこのシステムの使い方をすぐに学べると思う
8. このシステムは非常に使いにくいと思う
9. このシステムを使うことに自信が持てる
10. このシステムを使い始める前に多くのことを学ぶ必要があった

### スコア計算
- 奇数問: (回答 - 1)
- 偶数問: (5 - 回答)
- 合計 × 2.5 = SUSスコア（0-100）

### スコア解釈
- 80以上: 優秀
- 68以上: 平均以上
- 68未満: 改善が必要
```

### 6.3 NPSの実施と分析

```markdown
## NPS（Net Promoter Score）

### 質問文
「[サービス名]を友人や同僚に勧める可能性はどのくらいですか？」
（0: 全く勧めない 〜 10: 強く勧める）

### フォローアップ質問
- 0-6の回答者: 「どうすればより良くなりますか？」
- 9-10の回答者: 「特に気に入っている点は何ですか？」

### スコア計算
- 推奨者（9-10）の割合 - 批判者（0-6）の割合 = NPS

### セグメント別分析
| セグメント | 割合 | 主な理由 |
|-----------|------|---------|
| 推奨者（9-10） | % | |
| 中立者（7-8） | % | |
| 批判者（0-6） | % | |
```

## 7. インサイト抽出とレポート

### 7.1 分析フレームワーク

#### アフィニティダイアグラム

```markdown
## 分析手順

1. **データ抽出**
   - インタビューの発言、観察メモを付箋に書き出す
   - 1付箋1アイデア

2. **グルーピング**
   - 似た内容の付箋を近くに配置
   - ボトムアップでカテゴリを作成

3. **ラベリング**
   - 各グループに名前をつける
   - 上位カテゴリを作成

4. **パターン発見**
   - グループ間の関係性を分析
   - 重要なテーマを特定
```

#### インサイトの構造

```markdown
## インサイト記述フォーマット

### インサイト: [発見の要約]

**観察事実（What）**
[具体的に何が起きているか、データに基づく記述]

**背景・文脈（Context）**
[どのような状況で起きているか]

**ユーザーの動機（Why）**
[なぜそうなるのか、ユーザーの視点から]

**示唆（So What）**
[この発見がデザインや意思決定に与える影響]

**エビデンス**
- [引用1]
- [引用2]
- [データポイント]
```

## 8. リサーチ倫理とベストプラクティス

### 8.1 参加者への配慮

```markdown
## 同意取得チェックリスト

- [ ] 調査目的の説明
- [ ] 所要時間の明示
- [ ] 録音・録画の許可確認
- [ ] データの利用範囲の説明
- [ ] いつでも中止できることの説明
- [ ] 謝礼の説明
- [ ] 同意書への署名

## プライバシー保護

- 個人を特定できる情報は匿名化
- データは安全に保管
- 目的外利用の禁止
- 保管期限を設定
```

### 8.2 バイアスの認識と対策

| バイアス | 説明 | 対策 |
|---------|------|------|
| **確証バイアス** | 仮説を支持する情報を重視 | 反証を積極的に探す |
| **社会的望ましさ** | 良く見られたい回答をする | 匿名性の確保、行動観察 |
| **アンカリング** | 最初の情報に引きずられる | 質問順序をランダム化 |
| **ハロー効果** | 一部の印象が全体に影響 | 複数の評価軸を設定 |

## 9. クイックリファレンス

### 手法選択ガイド

```
質問: 何を知りたいか？

「ユーザーが何をしているか知りたい」
  → 行動観察、日記調査、アナリティクス

「なぜそうするのか知りたい」
  → ユーザーインタビュー

「どのくらいの人がそうか知りたい」
  → アンケート調査

「これで使えるか知りたい」
  → ユーザビリティテスト

「どの案が良いか知りたい」
  → A/Bテスト、コンセプトテスト
```

### サンプルサイズの目安

| 手法 | 推奨サンプルサイズ | 備考 |
|------|-------------------|------|
| ユーザビリティテスト | 5-8人 | 主要な問題の85%を発見 |
| インタビュー | 8-12人 | 飽和点まで |
| アンケート（探索的） | 100-300人 | |
| アンケート（検証的） | 400人以上 | 統計的有意性が必要な場合 |
| A/Bテスト | 統計的検出力による | 検出したい差分と有意水準で計算 |
